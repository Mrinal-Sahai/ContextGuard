# How CodeRabbit Works — Architecture Reference

> Compiled for ContextGuard development. Used as competitive benchmark and inspiration for the
> Semgrep + MCP integration strategy.

---

## Overview

CodeRabbit reviews PRs by running deterministic tools BEFORE any LLM is called, then feeding
structured ground-truth facts into the prompt. The LLM explains real findings — it does not guess.

```
GitHub Webhook
     │
     ▼
Ingestion Filter
  (~40% of PRs rejected: lockfile bumps, docs-only, formatting-only, bot commits)
     │
     ▼
Context Engine  ← runs BEFORE any LLM
  ├── Tree-sitter Code Graph     call graph, callers/callees across entire repo
  ├── Semantic Vector Index      (LanceDB) similar code retrieved by embedding
  └── Historical Learnings Store per-org rules extracted from past reviewer feedback
     │
     ▼
Deterministic Tools  ← results are FACTS, not inference
  └── 40+ tools:
      Security:    Semgrep, TruffleHog (secrets), Trivy (CVE), Checkov (IaC), OSV-Scanner
      Linting:     ESLint, Biome, Ruff, golangci-lint, PMD, RuboCop, ShellCheck, Hadolint
      Style:       Prettier, Black, gofmt
      Coverage:    lcov integration (if CI reports coverage)
      Complexity:  ast-grep patterns, radon (Python), gocyclo (Go)
     │
     ▼
Model Cascade
  ├── Fast classifier:  categorize each hunk (cosmetic / docs / logic / security)
  ├── Cheap model:      summaries, routine QA, formatting feedback
  └── Reasoning model:  o3 / o4-mini for logic bugs, security chains, cross-file impact
     │
     ▼
Agentic Verification  ← post-LLM, pre-post
  └── Before posting any comment, agent runs grep / ast-grep in sandbox to verify
      the issue actually exists at that exact location. Claims without evidence dropped.
     │
     ▼
Two-stage Noise Filter
  ├── Cheap LLM: filter redundant, low-confidence, or duplicate comments
  └── Confidence threshold: comments below threshold are silently dropped
     │
     ▼
Review posted to GitHub with line-level anchoring
```

## Why Their Output Is Specific (Not Generic)

### 1. Input format is semantic, not raw diff

CodeRabbit does NOT send a raw unified diff to the LLM. They send:
- A human-readable semantic summary of each changed file (generated by their own pre-pass)
- The actual method bodies extracted from the AST (not just diff lines)
- Linter findings already mapped to the exact line numbers in the diff

This is why their reviews say "at line 47 in `OrderRepository.java`, the JDBC query builds SQL
via string concatenation" rather than "there may be SQL injection concerns."

### 2. Linter results are injected as ground truth

The Semgrep finding `CWE-89 SQL injection at line 47` is a fact — the LLM just explains it in
plain English and suggests a fix. It cannot invent findings, and it cannot miss findings that
Semgrep already caught.

### 3. Code graph provides caller/callee context

When the code graph says "this method is called by `OrderService.checkout()` and
`CartService.apply()` — a change here has blast radius 2," that comes from the static graph,
not from the diff. The LLM knows the impact without having to infer it.

### 4. Agentic verification eliminates hallucination

Every claim in a generated comment is verified by a subprocess grep/ast-grep query before
posting. If the pattern doesn't exist at that location, the comment is discarded. This is the
primary reason CodeRabbit comments are specific and actionable — vague comments can't be verified
and get dropped.

## What ContextGuard Has That CodeRabbit Does Not

| Capability | CodeRabbit | ContextGuard |
|---|---|---|
| Research-backed quantitative scoring | No (rules-based thresholds) | **Yes** (Kim 2008, McCabe 1976, Banker 1993, etc.) |
| Reproducible defect probability formula | No | **Yes** — deterministic, citable |
| AST call graph diff (base vs head) | Yes (full repo graph) | **Yes** (PR-scoped) |
| SAST before LLM | Yes (40+ tools) | **Yes** (Semgrep OSS — 2000+ rules) |
| Sequence diagram of runtime flow | No | **Yes** (Mermaid sequenceDiagram) |
| Blast radius quantification | Partial (file-level) | **Yes** (module + layer + domain scope) |
| GitHub MCP (CodeQL alerts, CI status) | Via GitHub Actions bot | **Yes** (Anthropic MCP Connector) |
| Cognitive complexity scoring | Partial | **Yes** (McCabe + nesting penalty) |

## The Three Things That Close the Gap

Before the Semgrep + MCP integration, ContextGuard's gap with CodeRabbit was:

1. **No deterministic SAST before LLM** → LLM had to guess at security issues  
   Fix: `SemgrepAnalyzerService` runs `semgrep --config auto` on changed files before LLM call.
   Findings injected as ground-truth facts into the prompt.

2. **No actual code in the prompt** → LLM only saw diff statistics, not method bodies  
   Fix: `AIGenerationService.buildPrompt()` now includes a `STATIC_ANALYSIS_FINDINGS` section
   and `BEHAVIORAL_SIGNALS` section with actual annotation names, return types, and call edges.

3. **No real-time CI/security data** → LLM couldn't see CodeQL alerts or CI failures  
   Fix: `AnthropicAIClient` uses the MCP Connector beta to let Claude call the GitHub MCP server
   (`list_code_scanning_alerts`, `get_pull_request_reviews`, `get_pull_request_status`) during
   generation. Available when `LLM_PROVIDER=ANTHROPIC` and `GITHUB_TOKEN` are set.

## Key Architecture Insight

> CodeRabbit's "magic" is not the LLM. It's the pipeline that runs before the LLM.
>
> A senior engineer reviewing a PR has: the code in front of them, the test results, the linter
> output, the CI results, and knowledge of the codebase. CodeRabbit gives the LLM all of that.
> ContextGuard now does the same — with the additional layer of peer-reviewed quantitative scoring
> that CodeRabbit does not provide.
